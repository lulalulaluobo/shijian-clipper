# 项目需求总览 (PRD)

> 本文档只维护项目级目标、边界和里程碑。详细任务、决策、进度与工程规范统一由 Trellis 管理。

## 1. 项目背景与目标

做一个面向个人使用、低配置门槛的 Obsidian 转存工具：在 APK 中粘贴微信公众号文章链接，由 Cloudflare Worker 抓取文章、转换为 Markdown，并通过 Obsidian Fast Note Sync 写入本地库。

长期目标是把 APK 做成一个通用的“发送到我的 Obsidian”入口，支持图片、纯文本、网页以及其他可扩展的信息来源。

## 2. 核心用户与使用场景

- 主要用户：有自己的 Obsidian + Fast Note Sync 部署、希望在手机上快速收藏内容的个人用户。
- 第一阶段：粘贴一条微信公众号文章 URL，提交后看到成功/失败结果。
- 后续阶段：从系统分享菜单分享图片、文本或网页，一键转存到 Obsidian。

## 3. 功能范围

### 第一阶段

- APK：单一输入框、提交按钮、最近结果；设置页配置 Worker 地址、访问凭证、FNS 检测 token、Vault/目标文件夹。
- Worker：校验请求、抓取公众号文章 HTML、提取标题/作者/正文、转 Markdown，图片保留原始 URL，调用 FNS 写入。
- FNS：支持连接检测；返回写入路径和可读错误。
- 部署：单用户优先，支持个人自部署；敏感配置不写入 APK 包内。

### 第二阶段

- Android 系统分享入口。
- 纯文本、图片、普通网页等内容类型。
- 可选规则：按来源或内容类型选择目标文件夹。
- 可选服务端任务记录、重试和多用户部署模式。

## 4. 非目标与业务边界

- 第一阶段不做飞书、Telegram、公众号后台扫码、公众号订阅同步、AI 润色、管理后台和复杂账号体系。
- 第一阶段不把图片上传到图床；Markdown 只保存图片原链接，因此图片可访问性受原站策略影响。
- Worker 抓取公众号文章会受到页面结构变化、反爬、网络超时及 Cloudflare 执行限制影响；抓取器必须可替换，不能把这些问题伪装成稳定能力。

## 5. 技术约束与质量标准

- APK 采用 Android 原生 Kotlin，优先使用系统能力和少量依赖。
- 后端采用 Cloudflare Worker，保持无服务器和低运维；初期不引入数据库、队列或常驻 VPS。
- API 使用 HTTPS；token 仅在设置页输入并安全存储在设备端，Worker 不返回 token。
- URL、目标路径、响应大小和请求超时都在边界处校验。
- 文章转换和 FNS 调用必须可独立测试，失败时返回明确的阶段信息。

## 6. 里程碑

1. 方案确认：确定 APK 交互、Worker 鉴权、FNS API 兼容范围和部署方式。
2. 第一阶段 MVP：单篇公众号 URL → Markdown → FNS。
3. 真实设备验证：Android 分享/输入、失败重试、长文和图片链接。
4. 第二阶段扩展：系统分享入口与多种内容类型。
